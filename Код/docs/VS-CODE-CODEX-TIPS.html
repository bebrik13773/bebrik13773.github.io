Работа с задачами

Ставьте операционную задачу, а не общую просьбу

Вместо “исправь” лучше писать: найди причину, внеси правку, проверь diff, закоммить. Чем полнее задача, тем меньше холостых ответов.

Для больших задач задавайте рамки

Полезный формат: цель, ограничения, какие файлы можно трогать, что считать готовым.

Разделяйте “безопасный фикс” и “системное решение”

Если проект хрупкий, пишите без рефактора, только безопасный фикс. Если повторяется системно, пишите исправь системно, а не костылём.

Контекст в VS Code

  • Держите открытыми только нужные вкладки. Лишний шум мешает ориентироваться.
  • Перед сложной правкой полезно просить: сначала изучи структуру и только потом меняй.
  • Если файл большой, указывайте точную зону: смотри модалку настроек в pages/clicker/index.html и snapshot в api/state/sync.php.
  • Если баг плавающий, прикладывайте консольную ошибку, шаги воспроизведения, лог и скрин в одном сообщении.

Как писать запросы

Хороший базовый шаблон

Исправь баг. Не ломай X. Проверь связанные места. Сделай commit/push.

Для UI

Уточняйте: не редизайн, а точечный фикс или можно менять визуал, но без смены логики.

Для серверных правок

Сразу добавляйте: учти старую БД, старые данные и обратную совместимость.

Отладка

  • Самый сильный формат: вот ошибка → вот файл → вот лог → исправь и объясни точную причину.
  • Если баг воспроизводится только у пользователя, полезно просить: сделай защиту и fallback, даже если точная причина плавающая.
  • Если проект хрупкий, просите: сначала найди причину, потом внеси минимальный фикс.
  • После каждого нетривиального исправления полезно требовать: что изменено, где изменено, что проверить руками.

Фронтенд

  • Сразу уточняйте: desktop only, mobile first, не ломать touch.
  • Для тяжёлых экранов полезно писать: без тяжёлых blur/filter.
  • Для артефактов рядом с картинками сразу задавайте ограничение: используй img + object-fit, без background-image на bitmap-зонах.
  • Если нужен только UX fix, просите не трогать всю визуальную систему.

Рабочий поток

Панели VS Code

Держите открытыми Problems, Terminal и Source Control. Это экономит время после каждой правки.

Ищите связи до и после правки

Глобально ищите функцию, endpoint, storage key, CSS class и feature flag, чтобы не чинить только половину потока.

Делайте отдельные коммиты по темам

Так проще откатывать и понимать историю. Один баг — один коммит, если это возможно.

Для длинных проектов держите актуальными служебные файлы

AGENTS.md и roadmap-подобные страницы сильно ускоряют следующие проходы по проекту.

Готовые шаблоны запросов

Безопасный фикс

Исправь баг, сначала найди причину, потом внеси минимальный фикс. Без рефактора, не ломай связанные сценарии. После правки проверь diff и кратко напиши, что проверить руками.

Системная правка

Проблема повторяется в нескольких местах. Исправь системно, а не точечным костылём. Проверь клиент, сервер и все связанные UI-сценарии.

UI / UX fix

Нужен точечный UX fix. Не делай редизайн всей страницы, сохрани текущую логику и визуальный язык. Проверь desktop и mobile.

Фронтенд с тяжёлыми изображениями

Убери артефакты и лаги возле картинок. Используй безопасный img-рендер через object-fit, не возвращай background-image + blur/filter на bitmap-зонах.

Серверный фикс

Исправь серверный баг, учти старую БД и обратную совместимость. Если меняется schema/backfill, проверь все связанные endpoint-ы.

Большая задача

Сначала изучи структуру и свяжи все затронутые файлы. Потом внеси правки по шагам. Не останавливайся на анализе, именно внеси изменения и доведи их до рабочего состояния.
Лучший результат обычно получается не от “умных слов”, а от точной постановки: что сломано, что нельзя ломать, какие файлы вероятно связаны, что считать готовым.
